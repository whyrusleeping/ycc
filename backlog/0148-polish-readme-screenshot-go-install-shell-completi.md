## Description
Small first-impression polish, bundled:

- **README**: the repo contains a TUI screenshot that the README never shows — a docs-driven agent harness with a rich TUI should lead with a picture. Capture a *current* screenshot (the snapshot package can even render one deterministically) and reference it; add a `go install` line if the module path supports it.
- **Shell completions**: urfave/cli ships completion support; expose `ycc completion bash|zsh|fish` and document it. Session ids and project names are the high-value completions (`ycc attach <tab>`).
- **cli.md drift**: docs/cli.md's command table should be double-checked against the actual command tree once the above lands (spec-check covers doc paths, not flags).

## Acceptance criteria
- [ ] README shows a current TUI screenshot near the top; stale root-level screenshot removed or replaced.
- [ ] `ycc completion <shell>` emits a working completion script; documented in README + cli.md.
- [ ] `ycc attach` / `--project` complete against live sessions/projects when a daemon is reachable (best-effort).

## Plan

Three bundled polish items:

1) README screenshot (deterministic)
- Add an env-gated generator test in internal/tui (alongside TestSnapshotSessionView, reusing internal/tui/snapshot) that renders a *richer, larger* representative session frame (~110x32: user input, model turns, tool calls Read/Edit/Bash with results, a markdown-flavored idle report) and writes `docs/tui.png` when e.g. YCC_README_SCREENSHOT_DIR (or reuse YCC_TUI_SNAPSHOT_DIR pattern) is set. Plain `go test ./...` must stay write-free.
- Run the generator once, commit `docs/tui.png`, delete stale root `screenshot-2026-06-29-164952.png`.
- README: reference the image near the top (right after the intro paragraph), note it is generated by the snapshot renderer (regen command in a comment near the test or in the README alt text is fine).

2) Shell completions
- Root command: set `EnableShellCompletion: true` and use `ConfigureShellCompletionCommand` to un-hide the built-in `completion` command so it appears in `ycc --help` (urfave/cli v3 provides bash/zsh/fish/pwsh subcommands that emit scripts).
- Dynamic value completion, strictly best-effort and daemon-optional:
  - helper `completionClient()` that returns a client ONLY when `--addr` is set or `daemon.Reachable(daemon.LocalAddr, "")` — never starts the one-shot in-process daemon, never prints the stderr persistence notice; guard RPCs with a short (~1-2s) timeout context; on any error print nothing.
  - `attach` / `stop`: `ShellComplete` listing session ids via ListSessions (format `id:mode status` like cli's printCommandSuggestions `name:usage` convention, or plain ids — whichever renders cleanly in bash AND zsh).
  - commands with a `--project` flag (`start`, `cost`, `export`, `task` subcommands as practical): `ShellComplete` that, when the previous raw arg is `--project`, emits registered project names via ListProjects, else falls back to `cli.DefaultCompleteWithFlags`.
- Unit test: run the root command in-process with args `ycc completion bash` (and zsh/fish) capturing the Writer; assert a non-empty script mentioning `ycc` is emitted. Also assert `completion` shows in root help output.

3) Docs audit
- docs/cli.md: add a `ycc completion <shell>` section (source-in-shell examples) and the missing `ycc export` section (flags: --out, --full, --project); sweep the whole command table against the actual tree/flags in cmd/ycc/*.go and fix any other drift.
- README: add `go install github.com/whyrusleeping/ycc/cmd/ycc@latest` alongside the go build line; add a short "Shell completions" subsection.
- `go build ./... && go test ./...` clean; run `go run ./cmd/ycc spec-check` to confirm no stale doc refs (docs/tui.png referenced from README should exist before that).
